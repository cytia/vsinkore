## 三、动手前检查清单

每次写代码前逐项过一遍：

- **在哪一侧？** 这段逻辑属于 extension 宿主（Node、能碰文件系统 / VSCode API）还是 webview（浏览器沙箱、只能 postMessage）？两侧绝不共享运行时对象。
- **要不要跨进程？** 文件读写、图片落盘、读 VSCode 配置 / 主题 → 必须在 extension 侧做，webview 通过 postMessage 请求。
- **内核能不能直接给？** 编辑能力优先用内核已导出的 command / plugin / NodeView，不在插件里重写。先读内核 `src/index.ts` 导出面。
- **会不会触发回环？** 凡是 extension 回写 document 的路径，都要能识别"自己刚写的"，不再推回 webview。
- **参数超过 4 个吗？** 超过就先重新设计，改 options 对象。

## 四、代码规范

继承全局规范（英文代码与注释、注释只解释 why、删死代码、范围控制）。补充：

- **命名一致**：同一概念全仓用同一个词。内核已有词汇（`vaultRoot` / `saveImage` / `toRenderUrl` / `serializeMarkdown`）直接沿用，不另造同义词。
- **单一职责**：每个模块只做一件事；extension 侧 provider / 文件桥 / 图片桥分开，webview 侧 host wiring / UI / 消息处理分开。
- **每文件 ≤200 行**，超过拆分。
- **消息协议集中定义**：extension↔webview 的 postMessage 消息类型放单独文件，两侧共用同一份 TS 类型，不各写各的。

## 五、扩展宿主 ↔ Webview 边界（核心）

**单向数据流（对齐内核 / 桌面版约定）**
- webview 改动 → `serializeMarkdown` → postMessage → extension 用 `WorkspaceEdit` 整文替换 document。
- **保存 / 脏标记 / undo 栈交给 VSCode**，webview 侧不自存盘、不自管脏标记。
- 文档初始内容、外部改动由 extension 读 `document.getText()` 后 push 给 webview。

**防回环（必守）**
- extension 回写 document 会触发 `onDidChangeTextDocument`。必须识别这次变更是"插件自己刚写的"，跳过回推 webview，否则光标跳 / 死循环。

**消息边界**
- 只传 JSON 可序列化数据，不传函数、不传 DOM 引用、不传 EditorView。
- 每条消息带明确 `type` 字段；webview 收到未知 type 静默忽略。

## 六、Webview 安全（CSP / 资源 / 本地优先）

VSCode webview 默认强制 CSP，且不能直接 `file://`。永久规则：

- **脚本走 nonce**：CSP 用 `script-src 'nonce-xxx'`，每次生成新 nonce，不开 `unsafe-inline`。
- **静态资源走 `webview.asWebviewUri()`**：CSS / 图片 / 字体一律转 `vscode-webview://` 资源 URI；目录加进 `localResourceRoots`，不在白名单内的目录读不到。
- **第三方 CSS 打进 bundle**：katex / prosemirror 的 CSS 随 webview bundle 打包，不走 CDN（CSP 禁外域、且离线要可用）。
- **内容不出本地**：插件只读写工作区内的 `.md` 与其图片目录，不联网、不上传笔记内容。内核本就零网络，本插件不引入任何网络请求。
- **图片协议白名单**：渲染 `<img src>` 仅允许 `https:` / `data:image/*` / `asWebviewUri` 解析出的资源 URI，禁止 `javascript:` / 裸 `file://`。
- **外部 HTML 过滤**：若未来引入粘贴外部 HTML 的链路，须过滤后再进 ProseMirror（沿用内核已有处理，不放开 `<script>` / `on*=`）。

## 七、内核消费约定

- 内核**不带 CSS**：编辑器样式由本仓 `media/` 提供（移植桌面版编辑器 CSS + tokens，映射到 VSCode 明 / 暗主题）。
- 内核**不带 host wiring**：`createEditorState` 拿 `saveImage`；`EditorView` 的 `nodeViews` map 由本仓组装（参照内核 `example/main.ts`）。
- 内核改动需要新能力时，**优先在内核仓加导出**再消费，不在插件里 fork 内核逻辑。内核是独立私有仓，跨仓改动各自提交。
- 图片：`saveImage` 桥到 extension 写工作区图片目录返相对路径；`toRenderUrl` 经 extension 用 `asWebviewUri` 解析。

## 八、开发节奏

**Git 分支**
```
main      ← 稳定可运行
feat/xxx  ← 单个功能分支
fix/xxx   ← bug 修复分支
```

**Commit 格式**（沿用桌面版）
```
feat: 新增自定义文本编辑器骨架
fix: 修复回写 document 触发回环
chore: 升级 esbuild
```
不出现 AI 工具名或进度标签（`FIXED` / `Phase` / `Step` 等）。

**功能节奏**
- 按[阶段工作表](阶段工作表.md)五步逐个推进，不同时开多步。
- 每步完成后实际 F5 跑一遍验证，再进下一步。
