## 阶段零：方案敲定

### D0-1 · 形态 = Custom Text Editor（非独立面板）
注册为 `.md` 的 `CustomTextEditorProvider`，复用 VSCode 的文件生命周期（打开 / 脏标记 / 保存 / undo / 外部改动）。**否决独立 webview 面板**——那需自己实现打开哪个文件 / 保存 / 同步，工作量更大且与 VSCode 文件模型割裂。Custom Text Editor 最贴合内核"host 提供文件 IO、内核只管 Markdown↔ProseMirror"的定位。

### D0-2 · 付费边界 = 纯免费、只做编辑
插件首版只做 Markdown 编辑体验，**不碰激活 / PRO 导出 / 自定义字体主题**。理由：① 内核 scope 本就只含免费能力；② VSCode 沙箱无 Rust 验签层，反破解契约无法落地，强接 PRO 既不安全也偏离内核定位；③ 最快出可用版本。PRO 属于桌面版。

### D0-3 · 仓库 = 独立仓（内核引入方式已由 D-inline 取代）
独立仓 `D:\My-Projects\vsinkore`，与 `Inkore-core` / `My-MarkaV2` 平级，各自独立 git。**否决放进内核仓**（违背内核"不带 host 封装"定位）、**否决放进 v2 客户端仓**（会把 VSCode 依赖混进 Tauri 工程）。
**内核引入方式**：最初为 `file:../Inkore-core` 本地依赖，**现已改为源码内联**——见 D-inline。

### D-inline · 内核 = 源码内联进本仓，断开外部仓（结构契约）
`Inkore-core` 是独立组件，Inkore 本身不引用它；本仓也不再 `file:` 链接它，而是把内核 `src/*`（22 个 `.ts`）**拷贝内联**到本仓 `src/editor-core/`，内核的 15 个生产依赖收编进本仓 `package.json`。
**理由**：① 本仓应独立持有内核、可自由修改（含上轮搁置的脚注解析等），不受外部仓存在与否影响；② 去掉 `file:` symlink 后全仓单一 `node_modules`，D2-2 的双副本隐患连同 esbuild alias 一并消除，构建更简单。
**口径**：以本仓 `src/editor-core/` 为准；外部 `Inkore-core` 仅作源头记录、存量不动、不再同步。host wiring / CSS / 平台桥接仍在 `src/extension`、`src/webview`，与内核源码分目录隔离（内核仍不带 host 封装、不带 CSS）。
**验证**：`npm ls prosemirror-state` 全 deduped 单副本；`node esbuild.mjs` 双 bundle 通过；`tsc --noEmit` 零错误。

### D0-4 · 构建 = esbuild 双 bundle
extension 侧出 Node / CommonJS bundle，webview 侧出 browser bundle（含内核 + 自带 CSS）。两侧运行时隔离，绝不共享对象。选 esbuild 而非 tsup：VSCode 官方插件模板默认、最轻、双 target 配置直接。

### D0-5 · 定位 = 开发者文档场景 + 视觉对齐 VSCode（仍生效 · 设计契约）
插件用途收敛为"在 VSCode 里所见即所得地查阅 / 编辑项目文档、快速定位信息"，**不是把砚客桌面版编辑器整体搬进来**。因此**视觉完全对齐 VSCode**：配色用原生 `--vscode-*` CSS 变量（webview 天然注入）、字体跟随编辑器字体、明暗自动跟随主题，**不移植砚客的云霜 / 薄荷等主题**。排版结构（行距 / 块间距）可参考桌面版 CSS，但颜色 / 字体一律走 VSCode 变量。好处：主题切换零成本、融入 IDE 不突兀、省掉一套皮肤维护。

### D0-6 · 图片 = 只读显示已有，不支持插入（仍生效 · 范围契约）
首版图片只做**只读显示**：文档里已有的 `![](rel)` 经 `toRenderUrl` → extension `asWebviewUri` 解析渲染（"查阅项目文档"常带截图，必须能看到）。**不支持插入 / 粘贴 / 拖入新图**——`createEditorState` 必传的 `saveImage` 给 no-op（静默忽略或提示暂不支持插图），不写工作区、不碰图片落盘。内核 `imageUploadPlugin` 仍在 setup 里但回调不落盘。插入图片留待后续按需求再评估。

### D0-7 · 行号 = 左侧行号槽 + 源码行号口径（仍生效 · 功能契约 + 技术风险）
显示行号服务于"快速对齐信息 / 定位"。口径取**源码行号**（对齐 .md 文件真实行号，能和 VSCode 状态栏 / git diff / "第 N 行"对上），**非视图块序号**。形式为**左侧行号槽**（像代码编辑器，每个块对齐其起始源码行）。**技术风险**：WYSIWYG 块反查源码行是内核当前没有的能力——`parseMarkdown` 需把 markdown-it token 的 `map[0]`（源码起始行）带进 ProseMirror 顶层节点 attr 或维护映射表。可能要在内核加"带 sourceLine 的解析变体"，或在插件侧自建映射。阶段五动手前先做可行性验证，再决定改内核还是插件侧实现。

### D0-8 · 代码块语法高亮 = Shiki + 插件侧覆盖 NodeView（仍生效 · 功能契约 + 体积/CSP 坑）
**为什么是 vsinkore 重点而非内核重点**：砚客面向写作者，代码块只要能写能存；vsinkore 面向开发者看项目文档（README / API / 架构说明），代码块密集且是核心信息——IDE 里没有高亮 = 降级体验（旁边 VSCode 原生 .md 预览都有高亮）。**内核现状**：`CodeBlockNodeView` 有语言选择下拉 + 复制 + 删除，`<code>` 是纯文本、**无语法高亮、无行号**。

**方案**：
- **引擎 Shiki**（VSCode 同款 TextMate 语法 + 主题），高亮与 VSCode 像素级一致，并能吃 VSCode 当前主题色，与 `D0-5` 主题对齐联动（按明暗选 `dark-plus`/`light-plus` 等内置主题）。
- **插件侧覆盖 NodeView**：用 `prosemirror-highlight` + `prosemirror-highlight/shiki` 适配器自写 `code_block` NodeView，在 `nodeViews` map 覆盖内核的 `CodeBlockNodeView`。**内核零改动**（保持"高亮非内核重点"定位，符合 D0-4 两侧隔离）。
- **块内行号不做**：全局已有左侧源码行号槽（D0-7），代码块内不再套一层（口径不同会混淆）。

**已知坑（实现前看）**：
- **体积**：Shiki 全量语法包很大，webview **必须按需加载语言**（只 load 文档里实际出现的语言 + 懒加载），不能全量打进 bundle，否则启动慢。
- **CSP**：优先用 Shiki 的 **JS regex 引擎**（`@shikijs/engine-javascript`）避免 WASM，否则要放宽 webview CSP 加 `wasm-unsafe-eval`（与 D0-6 webview 安全收紧原则冲突）。

**实际落地（与原方案的偏差 · 已实现并 F5 验证）**：
- **不用 `prosemirror-highlight`、不覆盖 NodeView，改走 decoration**：ProseMirror 不允许往 contentDOM（可编辑 `<code>`）塞自建 span；自写高亮 plugin 用 inline `Decoration` 给文本上色（`src/webview/codeHighlightPlugin.ts`），内核 `CodeBlockNodeView` 的 header（语言下拉/复制/删除）原样保留，二者共存。仍是内核零改动。
- **CSP 合规上色 = token + nonce 授权 `<style>`**：Shiki token 只给 color hex 不给 scope class。host 侧把每个 hex 映射成 `.shk-<hex>` class、规则写进 extension 预置的 `<style nonce id="shiki-styles">`（`src/webview/shikiHighlighter.ts`），span 上无 inline style，**未开 `unsafe-inline`**。CSP 加 `style-src 'nonce-xxx'`。fontStyle 用固定 `.shk-i/.shk-b/.shk-u`。
- **按需加载靠 webview 改 ESM + 代码分割**：IIFE 不支持 splitting → 语言被全量打进主包（实测涨到 4MB）。改 `esbuild` webview 为 `format:"esm"` + `splitting:true` + `chunkNames:"chunks/[name]-[hash]"`，主包回落 1.68MB，~30 语言分片到 `dist/chunks/` 运行时按需 `import()`。HTML 入口改 `<script type="module">`，CSP `script-src` 加 `cspSource` 放行同源 chunk（nonce 只覆盖入口模块）。chunk 用相对 `./chunks/...`，在 `vscode-webview://` 下相对主模块 URL 解析，无需 publicPath。
- **主题跟随**：按 webview body 的 `vscode-light` class 选 `dark-plus`/`light-plus`，切主题时重建 highlighter + 清色表 + 重渲染。

## 阶段二：内核跑起来（PoC）

### D2-1 · 背景配色按 VSCode 面板/编辑器分层（细化 D0-5 · 设计契约）
对齐基准是本仓 `原型/editor-mockup.html`（全 `--vscode-*` 模拟 token，非砚客原型）。`[D0-5]` 说"配色走 VSCode 变量"，本条细化**正文区与代码块各用哪个底色**：
- **正文区底色 = `--vscode-panel-background`**（非 `--vscode-editor-background`）。开发者把插件当"文档查阅面板"，面板底色更贴该观感；与编辑器主底色拉开层次。
- **代码块体底色 = `--vscode-editor-background`**（比正文区略深一档，形成层次），header = `--vscode-editorWidget-background`。
- inline code 用 `--vscode-textPreformat-*`；表格四边框 `--vscode-widget-border`、表头 `--vscode-editorWidget-background`；H1/H2 下边框 `--vscode-textSeparator-foreground`；正文字体 `--vscode-font-family`、代码字体 `--vscode-editor-font-family`。
- **坑**：`--vscode-editorWidget-border` 不存在（会 fallback 透明 → 没边框），正确是 `--vscode-widget-border` / `--vscode-panel-border`。

### D2-2 · ProseMirror 必须单实例（契约 · 踩坑保留 · 修法已被 D-inline 取代）
**已不再适用 alias 修法**：内核源码内联进本仓后（见 D-inline），全仓只有一份 `node_modules`，双副本问题从根上消失，`esbuild.mjs` 的 alias 已删。`npm ls prosemirror-state` 验证全部 `deduped` 到单副本。

**踩坑保留作背景**：当内核曾经经 `file:` 引入时是 **symlink** 自带完整 `node_modules`，若本仓又直接装 `prosemirror-state`/`-view` 等会出现**两份物理副本**；ProseMirror 的 plugin key / schema 按对象等同性判定，双副本会让 `EditorState` 与插件不匹配，运行时报 **`Cannot read properties of undefined (reading 'localsInner')`**。教训：ProseMirror 全家桶必须单实例——内联后靠"单一 node_modules"保证，而非 alias。
**附带踩坑**：webview 内的 DOM/变量探测（`--vscode-*` 是否注入、body class）**必须切到 webview iframe 上下文**（devtools Console 顶部下拉选 `pending-frame (index.html)`）再跑；在主窗口 `top` 上下文跑查的是主窗口 body，会得到全空的假结果。

## 阶段三：双向桥

### D3-1 · 防回环 = 双层守卫（extension 文本比对 + webview apply 标志）+ 序列化延迟（契约 · 踩坑保留）
**防回环为何要两层而非一层**：单看 extension 侧，`lastWrittenText` 文本比对足以在源头拦掉"插件自己刚写的"change 事件、不回推 webview。但 webview 侧的 `applyExternal`（外部改动整文替换 doc）会 dispatch 一个 doc-changing transaction，**这个 transaction 同样触发 `dispatchTransaction` → `onChange` → 回传 `edit`**，绕一圈又把刚收到的外部内容当本地编辑写回去。故 webview 侧需 `applyingExternal` 标志：外部 apply 期间不序列化、不回传。两层守在不同 choke point，缺一不可（曾一度想删 webview 侧守卫，确认会引入多余回环往返，保留）。
**记录时机踩坑**：`lastWrittenText` 必须在 `applyEdit` **之前**记录——`onDidChangeTextDocument` 可能在 `applyEdit` 执行期间**同步**触发，若改成 `applyEdit().then()` 里记录，change 事件会先于赋值跑、比对落空、回环重现。故 `writeBack` 用 `onWritten` 回调在 apply 前同步记录（这层回调是承重的，非冗余）。
**序列化延迟进防抖**：`onChange` 传 `() => string` 序列化 thunk 而非已序列化字符串。webview 入口的防抖（200ms）只在定时器触发时调用 thunk——被合并掉的逐次按键不再各走一遍 `serializeMarkdown` 全树遍历，只有存活的那次编辑付代价。
**口径**：extension 整文替换用 `WorkspaceEdit.replace(fullRange)`，文本与当前 `getText()` 相同则 no-op（webview 往返序列化回同样 Markdown 时不脏标记）；外部同步走独立 `update` 消息（与初始 `init` 区分，webview 收到走 `applyExternal` 原地重渲染、不 remount，`addToHistory:false` 不进 undo 栈）。
