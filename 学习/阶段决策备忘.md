## 阶段零：方案敲定

### D0-1 · 形态 = Custom Text Editor（非独立面板）
注册为 `.md` 的 `CustomTextEditorProvider`，复用 VSCode 的文件生命周期（打开 / 脏标记 / 保存 / undo / 外部改动）。**否决独立 webview 面板**——那需自己实现打开哪个文件 / 保存 / 同步，工作量更大且与 VSCode 文件模型割裂。Custom Text Editor 最贴合内核"host 提供文件 IO、内核只管 Markdown↔ProseMirror"的定位。

### D0-2 · 付费边界 = 纯免费、只做编辑
插件首版只做 Markdown 编辑体验，**不碰激活 / PRO 导出 / 自定义字体主题**。理由：① 内核 scope 本就只含免费能力；② VSCode 沙箱无 Rust 验签层，反破解契约无法落地，强接 PRO 既不安全也偏离内核定位；③ 最快出可用版本。PRO 属于桌面版。

### D0-3 · 仓库 = 独立仓（内核引入方式已由 D-inline 取代）
独立仓 `D:\My-Projects\vsinkore`，与 `Inkore-core` / `My-MarkaV2` 平级，各自独立 git。**否决放进内核仓**（违背内核"不带 host 封装"定位）、**否决放进 v2 客户端仓**（会把 VSCode 依赖混进 Tauri 工程）。
**内核引入方式**：最初为 `file:../Inkore-core` 本地依赖，**现已改为源码内联**——见 D-inline。

### D-inline · 内核 = 源码内联进本仓，断开外部仓（结构契约）
`Inkore-core` 是独立组件，Inkore 本身不引用它；本仓也不再 `file:` 链接它，而是把内核 `src/*`（22 个 `.ts`）**拷贝内联**到本仓 `src/editor-core/`，内核的 15 个生产依赖收编进本仓 `package.json`。
**理由**：① 本仓应独立持有内核、可自由修改（含上轮搁置的脚注解析等），不受外部仓存在与否影响；② 去掉 `file:` symlink 后全仓单一 `node_modules`，D2-2 的双副本隐患连同 esbuild alias 一并消除，构建更简单。
**口径**：以本仓 `src/editor-core/` 为准；外部 `Inkore-core` 仅作源头记录、存量不动、不再同步。host wiring / CSS / 平台桥接仍在 `src/extension`、`src/webview`，与内核源码分目录隔离（内核仍不带 host 封装、不带 CSS）。
**验证**：`npm ls prosemirror-state` 全 deduped 单副本；`node esbuild.mjs` 双 bundle 通过；`tsc --noEmit` 零错误。

### D0-4 · 构建 = esbuild 双 bundle
extension 侧出 Node / CommonJS bundle，webview 侧出 browser bundle（含内核 + 自带 CSS）。两侧运行时隔离，绝不共享对象。选 esbuild 而非 tsup：VSCode 官方插件模板默认、最轻、双 target 配置直接。

### D0-5 · 定位 = 开发者文档场景 + 视觉对齐 VSCode（仍生效 · 设计契约）
插件用途收敛为"在 VSCode 里所见即所得地查阅 / 编辑项目文档、快速定位信息"，**不是把砚客桌面版编辑器整体搬进来**。因此**视觉完全对齐 VSCode**：配色用原生 `--vscode-*` CSS 变量（webview 天然注入）、字体跟随编辑器字体、明暗自动跟随主题，**不移植砚客的云霜 / 薄荷等主题**。排版结构（行距 / 块间距）可参考桌面版 CSS，但颜色 / 字体一律走 VSCode 变量。好处：主题切换零成本、融入 IDE 不突兀、省掉一套皮肤维护。

### D0-6 · 图片 = 只读显示已有，不支持插入（仍生效 · 范围契约）
首版图片只做**只读显示**：文档里已有的 `![](rel)` 经 `toRenderUrl` → extension `asWebviewUri` 解析渲染（"查阅项目文档"常带截图，必须能看到）。**不支持插入 / 粘贴 / 拖入新图**——`createEditorState` 必传的 `saveImage` 给 no-op（静默忽略或提示暂不支持插图），不写工作区、不碰图片落盘。内核 `imageUploadPlugin` 仍在 setup 里但回调不落盘。插入图片留待后续按需求再评估。

### D0-7 · 行号 = 左侧行号槽 + 源码行号口径（仍生效 · 功能契约 + 技术风险）
显示行号服务于"快速对齐信息 / 定位"。口径取**源码行号**（对齐 .md 文件真实行号，能和 VSCode 状态栏 / git diff / "第 N 行"对上），**非视图块序号**。形式为**左侧行号槽**（像代码编辑器，每个块对齐其起始源码行）。**技术风险**：WYSIWYG 块反查源码行是内核当前没有的能力——`parseMarkdown` 需把 markdown-it token 的 `map[0]`（源码起始行）带进 ProseMirror 顶层节点 attr 或维护映射表。可能要在内核加"带 sourceLine 的解析变体"，或在插件侧自建映射。阶段五动手前先做可行性验证，再决定改内核还是插件侧实现。

### D0-8 · 代码块语法高亮 = Shiki + 插件侧覆盖 NodeView（仍生效 · 功能契约 + 体积/CSP 坑）
**为什么是 vsinkore 重点而非内核重点**：砚客面向写作者，代码块只要能写能存；vsinkore 面向开发者看项目文档（README / API / 架构说明），代码块密集且是核心信息——IDE 里没有高亮 = 降级体验（旁边 VSCode 原生 .md 预览都有高亮）。**内核现状**：`CodeBlockNodeView` 有语言选择下拉 + 复制 + 删除，`<code>` 是纯文本、**无语法高亮、无行号**。

**方案**：
- **引擎 Shiki**（VSCode 同款 TextMate 语法 + 主题），高亮与 VSCode 像素级一致，并能吃 VSCode 当前主题色，与 `D0-5` 主题对齐联动（按明暗选 `dark-plus`/`light-plus` 等内置主题）。
- **插件侧覆盖 NodeView**：用 `prosemirror-highlight` + `prosemirror-highlight/shiki` 适配器自写 `code_block` NodeView，在 `nodeViews` map 覆盖内核的 `CodeBlockNodeView`。**内核零改动**（保持"高亮非内核重点"定位，符合 D0-4 两侧隔离）。
- **块内行号不做**：全局已有左侧源码行号槽（D0-7），代码块内不再套一层（口径不同会混淆）。

**已知坑（实现前看）**：
- **体积**：Shiki 全量语法包很大，webview **必须按需加载语言**（只 load 文档里实际出现的语言 + 懒加载），不能全量打进 bundle，否则启动慢。
- **CSP**：优先用 Shiki 的 **JS regex 引擎**（`@shikijs/engine-javascript`）避免 WASM，否则要放宽 webview CSP 加 `wasm-unsafe-eval`（与 D0-6 webview 安全收紧原则冲突）。

**实际落地（与原方案的偏差 · 已实现并 F5 验证）**：
- **不用 `prosemirror-highlight`、不覆盖 NodeView，改走 decoration**：ProseMirror 不允许往 contentDOM（可编辑 `<code>`）塞自建 span；自写高亮 plugin 用 inline `Decoration` 给文本上色（`src/webview/codeHighlightPlugin.ts`），内核 `CodeBlockNodeView` 的 header（语言下拉/复制/删除）原样保留，二者共存。仍是内核零改动。
- **CSP 合规上色 = token + nonce 授权 `<style>`**：Shiki token 只给 color hex 不给 scope class。host 侧把每个 hex 映射成 `.shk-<hex>` class、规则写进 extension 预置的 `<style nonce id="shiki-styles">`（`src/webview/shikiHighlighter.ts`），span 上无 inline style。fontStyle 用固定 `.shk-i/.shk-b/.shk-u`。（**注**：当初为此把 `style-src` 设为 `'nonce-xxx'`、刻意不开 `unsafe-inline`；该 CSP 口径后被 KaTeX 推翻，见 `[D5-7]`——Shiki 的 nonce `<style>` 在 `unsafe-inline` 下照样允许，本机制不受影响。）
- **按需加载靠 webview 改 ESM + 代码分割**：IIFE 不支持 splitting → 语言被全量打进主包（实测涨到 4MB）。改 `esbuild` webview 为 `format:"esm"` + `splitting:true` + `chunkNames:"chunks/[name]-[hash]"`，主包回落 1.68MB，~30 语言分片到 `dist/chunks/` 运行时按需 `import()`。HTML 入口改 `<script type="module">`，CSP `script-src` 加 `cspSource` 放行同源 chunk（nonce 只覆盖入口模块）。chunk 用相对 `./chunks/...`，在 `vscode-webview://` 下相对主模块 URL 解析，无需 publicPath。
- **主题跟随**：按 webview body 的 `vscode-light` class 选 `dark-plus`/`light-plus`，切主题时重建 highlighter + 清色表 + 重渲染。

## 阶段二：内核跑起来（PoC）

### D2-1 · 背景配色按 VSCode 面板/编辑器分层（细化 D0-5 · 设计契约）
对齐基准是本仓 `原型/editor-mockup.html`（全 `--vscode-*` 模拟 token，非砚客原型）。`[D0-5]` 说"配色走 VSCode 变量"，本条细化**正文区与代码块各用哪个底色**：
- **正文区底色 = `--vscode-panel-background`**（非 `--vscode-editor-background`）。开发者把插件当"文档查阅面板"，面板底色更贴该观感；与编辑器主底色拉开层次。
- **代码块体底色 = `--vscode-editor-background`**（比正文区略深一档，形成层次），header = `--vscode-editorWidget-background`。
- inline code 用 `--vscode-textPreformat-*`；表格四边框 `--vscode-widget-border`、表头 `--vscode-editorWidget-background`；H1/H2 下边框 `--vscode-textSeparator-foreground`；正文字体 `--vscode-font-family`、代码字体 `--vscode-editor-font-family`。
- **坑**：`--vscode-editorWidget-border` 不存在（会 fallback 透明 → 没边框），正确是 `--vscode-widget-border` / `--vscode-panel-border`。

### D2-2 · ProseMirror 必须单实例（契约 · 踩坑保留 · 修法已被 D-inline 取代）
**已不再适用 alias 修法**：内核源码内联进本仓后（见 D-inline），全仓只有一份 `node_modules`，双副本问题从根上消失，`esbuild.mjs` 的 alias 已删。`npm ls prosemirror-state` 验证全部 `deduped` 到单副本。

**踩坑保留作背景**：当内核曾经经 `file:` 引入时是 **symlink** 自带完整 `node_modules`，若本仓又直接装 `prosemirror-state`/`-view` 等会出现**两份物理副本**；ProseMirror 的 plugin key / schema 按对象等同性判定，双副本会让 `EditorState` 与插件不匹配，运行时报 **`Cannot read properties of undefined (reading 'localsInner')`**。教训：ProseMirror 全家桶必须单实例——内联后靠"单一 node_modules"保证，而非 alias。
**附带踩坑**：webview 内的 DOM/变量探测（`--vscode-*` 是否注入、body class）**必须切到 webview iframe 上下文**（devtools Console 顶部下拉选 `pending-frame (index.html)`）再跑；在主窗口 `top` 上下文跑查的是主窗口 body，会得到全空的假结果。

## 阶段三：双向桥

### D3-1 · 防回环 = 双层守卫（extension 文本比对 + webview apply 标志）+ 序列化延迟（契约 · 踩坑保留）
**防回环为何要两层而非一层**：单看 extension 侧，`lastWrittenText` 文本比对足以在源头拦掉"插件自己刚写的"change 事件、不回推 webview。但 webview 侧的 `applyExternal`（外部改动整文替换 doc）会 dispatch 一个 doc-changing transaction，**这个 transaction 同样触发 `dispatchTransaction` → `onChange` → 回传 `edit`**，绕一圈又把刚收到的外部内容当本地编辑写回去。故 webview 侧需 `applyingExternal` 标志：外部 apply 期间不序列化、不回传。两层守在不同 choke point，缺一不可（曾一度想删 webview 侧守卫，确认会引入多余回环往返，保留）。
**记录时机踩坑**：`lastWrittenText` 必须在 `applyEdit` **之前**记录——`onDidChangeTextDocument` 可能在 `applyEdit` 执行期间**同步**触发，若改成 `applyEdit().then()` 里记录，change 事件会先于赋值跑、比对落空、回环重现。故 `writeBack` 用 `onWritten` 回调在 apply 前同步记录（这层回调是承重的，非冗余）。
**序列化延迟进防抖**：`onChange` 传 `() => string` 序列化 thunk 而非已序列化字符串。webview 入口的防抖（200ms）只在定时器触发时调用 thunk——被合并掉的逐次按键不再各走一遍 `serializeMarkdown` 全树遍历，只有存活的那次编辑付代价。
**口径**：extension 整文替换用 `WorkspaceEdit.replace(fullRange)`，文本与当前 `getText()` 相同则 no-op（webview 往返序列化回同样 Markdown 时不脏标记）；外部同步走独立 `update` 消息（与初始 `init` 区分，webview 收到走 `applyExternal` 原地重渲染、不 remount，`addToHistory:false` 不进 undo 栈）。

## 阶段四：图片只读显示 + 主题对齐

### D4-1 · 图片渲染桥 = extension 算 base + webview 拼接；图片根 = vault 根（契约 · 口径修正踩坑）
**为何 base 在 extension 算**：内核 `ToRenderUrl = (vaultRoot, relPath) => string` 是**同步**函数，但相对路径 → `vscode-webview://` 资源 URI 只有 extension 能解析（`asWebviewUri`，webview 沙箱碰不到）。故 extension 在 `resolveCustomTextEditor` 时一次性算好 base（`asWebviewUri(imageRoot).toString()`），随 `init`/`update` 下发；webview 侧 `toRenderUrl` 用 `new URL(relPath, base)` 纯字符串拼接（同 `[D0-8]` chunk 的相对解析思路）。base 算一次、编辑器生命周期内不变。
**口径修正（踩坑）**：最初取 **文档所在目录** 为图片根（本会话曾据此拍板"文档所在目录"），子文件夹内的笔记图片全部 "图片加载失败"。根因：Inkore vault 惯例是 **单个 `.images/` 放 vault 根**、任意深度的笔记都写 `![](.images/x.png)` 引用它；以文档目录为根会把 `.images` 拼到 `子目录/.images/`（不存在）。**URL 拼接逻辑本身没错**（headless 验证过），错的是 base 目录口径。**修正**：图片根改取 `workspace.getWorkspaceFolder(document.uri)` 即 vault 根，文档不在任何 workspace 时回退文档目录；`localResourceRoots` 同步授权 vault 根。**覆盖**之前"文档所在目录"的决定。
**最小复现**：`vault/.images/x.png` + `vault/sub/note.md` 内 `![](.images/x.png)` → 打开 note 显示"图片加载失败"；把 note 移到 `vault/note.md` 即正常（隔离出子目录偏移）。
**范围 / 安全**：`saveImage` 维持 no-op（插图静默忽略，`[D0-6]`）；图片协议白名单——内核 `URL_RE` 只放行 `http(s):`，其余走 `toRenderUrl` 拼成受控 `vscode-webview://` URI，裸 `file:`/`javascript:` 不被放行；CSP `img-src` 含 `cspSource https: data:`，vault 根图片经 `asWebviewUri` 是同源 URI，无需放宽。
**主题对齐**：阶段二 `media/editor.css` 已全 `--vscode-*` 明暗自动跟随；本阶段只补图片 + 加载失败态样式（`.pm-image` / `.pm-image-error`，沿用同款变量），未引入新主题机制。

## 阶段五：行号槽 + 交互打磨

### D5-1 · 行号槽可行性 = 插件侧平行解析取源码行，内核零改动（可行性结论 · 技术风险关闭）
`[D0-7]` 标的技术风险——"WYSIWYG 块反查源码行是内核没有的能力"——headless 验证后**确认可行，走插件侧、内核零改动**。
**验证发现**：用内核同款配置的 markdown-it 对文档文本解析，**每个顶层块都有对应的 block-start token 且携带 `token.map = [startLine, endLine]`（0-based）**；该 token 序列与 `parseMarkdown` 出的 ProseMirror doc 顶层节点**顺序 1:1 对应**。故按 index 位置 zip（第 i 个 token 的行号 → 第 i 个顶层节点）即得每块源码起始行，再用 `view.coordsAtPos(nodePos)` 在左侧槽对齐渲染行号。
**否决"改内核"路线**：让 `MarkdownParser`/`tokenMap` 把 `map[0]` 写进每个节点的 `sourceLine` attr，要动内核 schema + 每种块节点类型，侵入大、且违背"内核不带行号"立场（`[D0-7]`/`[D0-8]` 同源定位：高亮/行号非内核重点）。插件侧平行解析自包含在 webview，与该立场一致。
**踩坑（实现前必看）**：顶层块过滤口径是 `token.level === 0 && token.nesting >= 0 && token.map`——`fence`(代码块)/`hr`(分割线)/`html_block` 是 `nesting: 0`（自包含块）而非 `nesting: 1`（开标签），若只过滤 `nesting === 1` 会漏掉代码块等、导致行号与节点错位。
**实例来源**：host 侧平行解析需用**内核导出的、配置好的 `mdIt` 单例**（`markdown.ts:124-129` 应用 katex/task/htmlInline/tableCellWrap/footnote 插件后的实例），保证分块口径与渲染完全一致；host 自建 `new MarkdownIt()` 会因缺这些插件（如 task_list 被当普通 list）分段不一致、行号错位。导出点在 `src/editor-core/index.ts` 加一行 `export { mdIt }`（实现留待动手轮）。
**实现落地（已 F5 验证）**：内核 `index.ts`/`markdown.ts` 导出配置好的 `mdIt`；host 侧 `src/webview/sourceLines.ts` 的 `collectBlockLines(text)` 按上述口径取每块源码起始行；`src/webview/lineGutter.ts` 是**绝对定位兄弟层**（非 PM decoration——PM 不许往可编辑 contentDOM 塞 widget，同 `[D0-8]`），叠在 `#root` 滚动容器内、`.ProseMirror` 左侧 lane（`#root` 左 padding 留槽宽，正文 `max-width` 居中不动），每块用 `coordsAtPos(offset+1)` 定位画右对齐行号、无竖线。重算 rAF 节流——它内部要 `serializeMarkdown` 取文本做映射，靠 rAF 保证每帧至多一次（非逐键）；`docChanged`/滚动/resize 触发。
**空行行号 = 放弃（讨论结论）**：曾想让空行/块内续行也显示行号（淡色）以消除"跳号像算错"的观感。但 WYSIWYG 下空行被渲染成块间 margin、无可 `coordsAtPos` 对齐的视觉行，段落源码 1 行可能视觉换多行——"每物理行画号并对齐"在所见即所得里无处落点。近似方案（块顶堆叠空行号 / 跳号处填淡点）增复杂度且非真连续行号，**确认不做**，保持"逐块标首行 + 源码物理行号"，与 .md 真实行号对齐即够。
**口径自洽性**：跳号（1,3,5,7…）是块间空行的真实体现、非计数错误——headless 核对 `collectBlockLines` 输出与源码物理行号逐一吻合。源码物理行号口径下"只标块首行 + 跳号"是自洽的（若改块连续序号才与跳号矛盾，但口径定为源码行号 `[D0-7]`）。
**修正（空段落破坏 1:1 zip · 踩坑）**：上面"PM 节点数与行号数 1:1"的假设**对空段落不成立**。WYSIWYG 里空行是一个可编辑的空 `<p>` 顶层节点，但 markdown-it **不把空行解析成块**（headless：`的\n\n\n\n的` → PM 3 个顶层节点、markdown-it 仅 2 块）。原 `lineGutter` 按 index 直接 zip，遇空段落即整体错位、末块拿 `undefined` 丢号（现象：行号偏移 + 最后一块无号）。**修法**：`rebuild`/`reposition` 遍历 `doc.forEach` 时跳过空段落（`type==='paragraph' && content.size===0`）——不消耗行号 index、不画号，使"有内容的块"与 markdown-it 块重新 1:1 对齐。空段落本就不该有号（同"空行不标号"口径）。

### D5-2 · 查找框 = webview 内自建 + 内核 searchHighlightPlugin 驱动（契约 · 范围）
**为何不复用 VSCode 原生 find**：webview 是独立 iframe，VSCode 编辑器的 Ctrl+F find widget **够不到 webview 内部 DOM**，无法高亮 ProseMirror 文档里的匹配。故查找 UI 必须在 webview 内自建。
**接法（内核零改动）**：内核已导出 `searchHighlightPlugin`（默认在 setup plugins 里，decoration 上 `.search-match`/`.search-match-active`）+ `searchHighlightKey`。host 侧 `src/webview/searchBox.ts` 是绝对/固定定位的查找框，靠 transaction `setMeta(searchHighlightKey, {query, opts, activeIndex})` 驱动内核 rescan + 重绘；内核经 `createEditorState({onMatchesChange})` 回调把匹配列表抛回 host 更新计数（`editorHost.ts` 透传该回调）。沿用内核既有 `SearchOpts`/`SearchMatch` 词汇，不另造。
**触发**：`index.ts` 在 window `keydown` **capture 阶段**截 `Ctrl/Cmd+F`（抢在 VSCode 原生 find 前；反正它够不到 iframe），`preventDefault` 后 `toggleSearch`；`Esc` 关闭并清查询 + 清高亮。
**导航不动文档选区**：next/prev 用 `view.domAtPos(match.from)` 的 DOM `scrollIntoView` 滚到匹配，**不设 ProseMirror 选区**——否则光标会被从查找输入框抢走。activeIndex 改动走 activeIndex-only meta（内核 `apply` 里 query 未变不 rescan、不触发 `onMatchesChange`，无回环）。
**定位坑**：查找框必须 `position: fixed` 而非 `absolute`——`#root` 是 `overflow-y:auto` 滚动容器，absolute 会让查找框随内容滚走；fixed 钉在 webview 视口右上角（同 VSCode 原生 find 钉法）。
**范围（首版止于此）**：只做高亮 + 导航，**不做替换**（find-and-replace 不在工作表"查找高亮"范围）、**不做 vault 全局查找**（`SearchOpts.vaultScope` 仅透传不实现）。Aa（caseSensitive）/ .*（regex）选项接内核已支持的口径；fuzzy 内核支持但未在 UI 暴露。CSP 无需放宽（无新脚本、无外域）。

### D5-3 · 空态走内核占位 / 加载态 host 预置 / priority=default（契约 · 范围）
**空态不重做**：空文档占位由内核 `placeholderPlugin` 已给（唯一空段落时 widget decoration `.pm-placeholder`，ghost-text 配色已明暗跟随）。本插件沿用，不另造空态——`createEditorState({placeholder})` 即用内核能力，符合"内核能给的不在 host 侧重写"。
**加载态 = HTML 预置静态占位**：webview 脚本加载到收到 `init` 之间 `#root` 空白。在 provider 的 `buildHtml` 里把 `#root` 初值设为 `<div class="pm-loading">Loading…</div>`（**静态 HTML，不依赖脚本**——连脚本执行前的空窗也覆盖）；webview 收 `init` 时已有的 `mount.textContent=""` 自动抹掉占位、无缝切编辑器。纯 host 侧，无消息协议改动、无内核改动。不加 spinner/keyframes（一句淡色文案够，避免引入动画资源）。
**`customEditors.priority` = `option` → `default`**：原 `option` 下 `.md` 默认仍由 VSCode 内置文本编辑器打开，必须右键 Reopen With 才进 vsinkore。改 `default` 后打开 `.md` 直接进 vsinkore（用户仍可 Reopen 回 Text Editor）。**坑**：`contributes` 改动要 **F5 重起扩展开发宿主**才生效（激活时读注册表），reload window 不够。

### D5-4 · Bubble Toolbar = host 自写浮层 + 内核定位/command（契约 · 范围 · 设计）
**接法（内核零改动）**：内核 `bubbleToolbarPlugin(onSelectionChange)` 只在选区变化时回调 `BubbleSelectionInfo | null`（选区几何 + 卡片 rect），host `src/webview/bubbleToolbar.ts` 自写浮层、调内核 `calcBubblePosition(info, w, h)` 算位置（上/下翻转 + 贴卡片边 clamp）、接内核 command（`toggleBold/Italic/Underline/Strikethrough/Code`、`toggleTextColor`、`upsertLink`）。激活态 `.on` 由 `isMarkActive` 驱动。`editorHost.ts` 给 `createEditorState` 传 `onSelectionChange`（触发内核把 `bubbleToolbarPlugin` 加进 plugins）。
**UI 严格复刻原型**：照搬 `原型/editor-mockup.html` 的 `.bubble`（类名/结构/CSS 逐行搬）——8 项 B/I/U/S/`</>`/分隔/单色 swatch/🔗。**无箭头三角**（原型本就没画）。
**文字色只一个（范围）**：不做桌面版那排色板。swatch **显示**色用 `--vscode-charts-green`（随主题；原型用的紫 `charts-purple` 与 VSCode 调性不搭、且易和链接蓝混淆，改绿——语义中性、与链接/选中蓝拉得开）；但**写进文档的 mark 色必须固定 hex** `#4ec9b0`（`toggleTextColor` 收字面 hex 写进 `textColor` mark→序列化进 `<span style>`，**不能存 CSS 变量**，否则换主题/换编辑器失真、且非法颜色破坏序列化）。两层分离是硬约束。heading 下拉的 active 标记同步用 `charts-green` 保持一致。
**保选区**：按钮用 `mousedown`+`preventDefault` 触发 command（click 会先 blur 掉选区）；内核 blur 守卫认 `[data-bubble-toolbar]` 属性，故浮层根须带它。链接走 `window.prompt` 收 URL（MVP 不自写链接输入浮层，原型也没画该态）。
**浮出动画**：查证 VSCode 编辑器浮层（suggest/hover widget）**本身不带浮出动画**（直接显隐）；选用 VSCode 其他浮层同款全局 `fadeIn`（`opacity 0→1`，~0.15s ease，**无位移**）。bubble 用 `hidden` 显隐，`display:none→显示`自动重放 animation，JS 零改动。
**tooltip = 自绘 VSCode hover 同款**：原生 `title` 灰、慢、不可控。通用 `src/webview/tooltip.ts`：元素改用 `data-tooltip`（去原生 title），事件委托单例浮层、悬停 ~500ms 显示、配色 `--vscode-editorHoverWidget-background/-foreground/-border`、`position:fixed`+视口 clamp；bubble 与查找框共用（`installTooltips()` 幂等）。

### D5-bug · Bubble 延迟根因 = 异步语言加载逐个 RECHECK × 同步全量 tokenize 阻塞主线程（踩坑 · 修法契约）
**现象**：含多代码块的文档里，框选松手后 bubble 延迟 0.4~0.8s 才现（`[D5-4]` bubble 定位本身只 0~4ms，无辜）。
**复现 + 定位**：临时计时打点证实——`mouseup` 后内核 `setTimeout(tryShow, 10)` 的 **10ms 定时器实际延迟 400~820ms 才触发**（主线程被占），而同期 `codeHighlightPlugin` 的 `buildDecorations` 连发十几条、单条 18~483ms，全部 `recheck:true docChanged:false`。
**根因**：代码块语法语言**逐个异步加载完成**时各 dispatch 一次 `RECHECK`，每次 `buildDecorations` **同步全量重 tokenize 整篇所有代码块**（Shiki JS 正则引擎，O(语言数 × 全文代码量)），这串同步重建阻塞主线程，把 bubble 的 10ms 定时器挤后数百 ms。
**修法 ①（治本 · tokenize 缓存）**：`shikiHighlighter.ts` 的 `tokenize` 加 `(activeTheme, lang, code)` key 缓存（bound 256 条 FIFO），未变块复用上次 token 结果。RECHECK 连发时只有新加载语言的块真跑 Shiki，其余近零成本。主题切换 `resetColorRules` 一并清缓存。
**修法 ②（削峰 · RECHECK 合并）**：`makeLangRequester` 把多语言加载完成的多次 RECHECK 用 **`requestAnimationFrame` 合并成一次**（独立 `import()` 在错开 tick resolve，microtask 收不拢，须以帧为窗口）。回调内 `view.isDestroyed` 守卫。
**口径**：① 治本（消除重复 tokenize）、② 削峰（收敛 dispatch + 重建次数），叠加。两步均 F5 验证 bubble 松手即现、首开高亮一次性上色。

### D5-5 · 右键菜单 = host 自建单层菜单 + 链接浮层共享（契约 · 范围 · 结构）
**接法（内核零改动）**：内核只导出 `ContextMenuCoords` 类型、无现成菜单 plugin，故 host 侧自建（同 bubble/查找框定位）。`src/webview/contextMenu.ts` 监听 `view.dom` 的 `contextmenu`（`preventDefault` 抑制原生菜单）、在鼠标位置弹出（视口 clamp），点项派发内核 command、Esc/外部 mousedown 关闭，菜单根带 `data-bubble-toolbar` 复用内核 blur 守卫。
**结构（对齐 Inkore 原版，非本仓原型占位）**：**单层菜单 + 三分组标题**（格式 / 插入 / 编辑），唯一二级是"表格"行列浮层（`contextMenuTable.ts`，默认 3×3 → `insertTable(r,c)`）。**无 Cut/Copy/Paste 原生剪贴板项、无快捷键标签**（对齐截图）；原生剪贴板交给键盘快捷键。
**范围裁剪**：截图含"图片"项，但本插件 `saveImage` 是 no-op（`[D0-6]` 图片只读、不插入），放一个点了没反应的项会误导，**去掉图片项**。
**链接浮层共享（小重构）**：bubble 的链接输入浮层（`mountLinkInput`）由 `bubbleToolbar` 内部创建**上提到 `editorHost`**，作为单实例传给 bubble 和右键菜单两处共用，避免两份。`mountLinkInput` 随之解耦：自带 `runCmd`（不再吃 bubble 的 `refreshActive`）、`open(anchor: DOMRect)` 按容器 rect 定位（支持从 bubble 链接按钮或菜单项两个锚点打开）。
**右键命中节点的让位**：菜单挂 `view.dom`，内核图片/公式/目录 NodeView 各有自身 contextmenu 处理并 `stopPropagation`——右键这些节点走它们自己的菜单、不弹格式菜单，符合预期（非冲突）。
**表格内 = 另一套独立菜单（非子菜单切换）**：`contextMenu.ts` 在 `contextmenu` 事件用内核 `isInTable(view.state)` 分发——光标**在表格外**弹格式菜单（含"表格 ›"插入新表格行列浮层）；**在表格内**弹独立的表格编辑菜单（`contextMenuTableEdit.ts`：行/列/单元格三组 + 删除表格，对齐 Inkore 原版截图），接内核 `addRow/Column Before/After`、`deleteRow/Column`、`mergeCells`、`splitCell`、`deleteTable`。两套**不同时出现**，故表格编辑项天然都可用、无需禁用态。表格编辑菜单无图标列（`.ctxmenu--plain`，截图无图标）。

### D5-6 · 表格内精简 bubble = 同组件 compact 变体 + isInTable 分发（契约 · 范围）
表格单元格里用**普通文本选区**选词时，内核 `bubbleToolbarPlugin` 照常回调（仅 `CellSelection`/整行整列选中时内核自身返 null 不显示）。要求此时弹**精简 bubble**——只 7 项：加粗/斜体/下划线/删除线/行内代码/标色/链接，**不带块级**（标题/引用/列表/复制/清除格式，单元格里无意义）。
**实现（同组件 + options.compact，非另起文件）**：`mountBubbleToolbar` 加 `{ compact }` 选项；compact 下只建 marks + 标色 + 链接，跳过 heading 菜单/块切换/复制清除。定位、refresh、blur 守卫等逻辑两变体共用，避免重复一套。标色（绿 swatch + 入档 `#4ec9b0`）与链接（共享 `mountLinkInput` 实例）与正文 bubble**完全一致**。按钮原语（`ToggleButton`/`markActive`/`blockActive`）抽到 `bubbleButtons.ts` 共用（兼顾 200 行/文件）。
**分发（editorHost）**：`onSelectionChange` 接一个 `dispatchBubble`——有选区且 `isInTable(view.state)` 则显 compact bubble、隐正文 bubble；否则反之。两个 bubble 实例**不同时显示**。内核零改动（沿用现成 `bubbleToolbarPlugin` 回调 + `isInTable`）。

### D5-7 · KaTeX 公式塌 = CSP `style-src` 缺 `unsafe-inline`（踩坑 · 修法 · 取代 D0-8 CSP 口径）
**现象**：复杂公式（`\frac`/上下标/`\sum`/`\int` 上下限）渲染塌掉——字形/字号在，但靠定位的上下标、分式线全跑位。
**定位（运行时证据）**：webview devtools Console 海量 `Applying inline style violates ... 'style-src' ...`，调用栈 `renderTex @ webview.js`。即 **KaTeX 渲染产物用逐元素 inline `style`**（`.vlist` strut 高度、margin、vertical-align 全靠 inline style 定位），而 CSP `style-src` 当时是 `${cspSource} 'nonce-${nonce}'`、**不含 `unsafe-inline`**，KaTeX 的每条 inline style 被 CSP 拦掉 → 定位全失效。静态侧（CSS 打包、字体、`font-src`、KaTeX reset）全对，纯 CSP 拦截。
**根因关系**：与 `[D0-8]` 给 Shiki 上色时"刻意不开 `unsafe-inline`、改 nonce `<style>` 色表"同源约束；但 Shiki 色表是**我们自己可控**的、能改成 class+nonce，KaTeX 是第三方库、产物 inline style **动态且无穷**（每个公式不同），nonce/hash 都覆盖不了。
**修法**：`style-src` 改为 `${cspSource} 'unsafe-inline'`，**去掉 style 的 nonce**（CSP 规范：`style-src` 一旦含 nonce/hash，`'unsafe-inline'` 即被浏览器忽略，二者不能并存）。Shiki 的 `#shiki-styles` nonce `<style>` 在 `unsafe-inline` 下照样允许（同源 `<style>`），机制不受影响；`#shiki-styles` 的 `nonce` 属性随之去掉（已成死属性）。**`script-src` 的 nonce 不动**——脚本侧收紧不放松，只 `style-src` 放宽。对齐 VSCode 官方 markdown 预览（其 KaTeX/highlight 也用 `style-src 'unsafe-inline'`）。**取代** `[D0-8]` 的 "未开 unsafe-inline / style-src 用 nonce" 口径。
